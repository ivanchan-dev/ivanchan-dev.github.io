The problem being solved
What this build does

Today, lease deal data lives in spreadsheets and is manually uploaded to Anaplan. There is no single system of record for the full lease lifecycle — from board approval through to a signed contract, site opening, and eventual renewal or termination.

This build creates that system in Salesforce, integrating with the existing Anaplan Centre Module and the Finance rent schedule import process. The Salesforce Lease Opportunity becomes the master record for all deal data. Everything downstream — the site, the signed contract, the rent data — is created automatically by flows.

Option B decision (critical): The old design used an intermediate object called OpportunitySite__c to link the deal to the site before signing. That object is eliminated. Instead, LeaseSite__c is created earlier (at Board Approved stage) with Status = Draft, and is promoted to Active when the lease is signed. This means Finance can import rent schedules against a Draft record using exactly the same Data Loader process they use today — no workflow change for Finance.

Anaplan integration
Salesforce → Anaplan Sync Rules
One-way sync. Salesforce is the source of truth. Site__c is the sync source — one Anaplan row per site, always.
Rule 1 — Freeze gate: Always check Anaplan_Freeze__c BEFORE querying data to aggregate. If TRUE, skip the record entirely. Do not send zeroes — that would corrupt Anaplan history.
Rule 2 — Draft exclusion: Only Active LeaseSite__c records contribute to Site__c aggregation. Filter Status__c = 'Active' in all roll-ups and the Lease Expiry Tracking. Draft records must never appear in Anaplan.
Rule 3 — Multi-landlord: When Site__c.Multi_Lease__c = TRUE, only the Lease__c where Flow_To_Anaplan__c = TRUE contributes financial data to the sync.
Rule 4 — Freeze lifecycle: Flow 05 sets freeze = TRUE when the last Active LC expires. Flow 06 sets freeze = FALSE when a new LC becomes Active. The sync resumes automatically — no manual intervention.

Things that will break
Watch-outs & Common Mistakes
These are the issues most likely to cause problems during build. Most of them are not obvious until something breaks in testing.
⛔ LeaseSite__c — Opportunity__c must be Lookup
The Opportunity link on LeaseSite__c must be a Lookup, not a Master-Detail. Site__c already occupies the one Master-Detail parent slot allowed per object. If you create it as Master-Detail, you'll hit a validation error. The consequence of using a Lookup is that you lose cascade delete — Flow 02 handles cleanup explicitly.
⛔ Flow 12 governor limits
A 37-row Finance import fires Flow 12 thirty-seven times. Each execution deletes all existing LeaseRentSchedule__c rows and recreates them. At scale this will hit DML limits. Do not build this as a simple record-triggered flow. Use a Platform Event or Apex batch to consolidate all Monthly_Lease__c saves in one import batch into a single Flow 12 execution per LeaseSite__c.
⛔ Draft exclusion from aggregates
Every roll-up summary and formula on Site__c that reads from LeaseSite__c must filter Status__c = 'Active'. Draft records must never contribute to area totals or Anaplan data. If you forget this, Finance's preliminary import will corrupt the site's Anaplan figures before the lease is even signed.
⛔ Anaplan freeze timing
The Anaplan sync must check Anaplan_Freeze__c before querying aggregation data — not after. If you query first and then check the flag, you'll send stale data to Anaplan during a gap period. This corrupts historical Anaplan data which is very hard to fix retroactively.
⚠ Rent_Free_Value__c is negative
Finance stores rent-free months as negative values in Rent_Free_Value__c (e.g. −326,222). Flow 12 identifies rent-free periods by counting rows where Rent_Free_Value__c < 0. If Finance ever imports positive values here the rent schedule derivation will produce incorrect results. Add a validation rule.
⚠ Field API names cannot be renamed
The 15 "RELABEL" fields in the data model Excel cannot have their API names changed. Only the field label can be updated. Any reports, flows, validation rules, or integrations referencing the old API name will continue to work — the label change is purely cosmetic for the UI.
⚠ Flow 09 — Before + After Save pattern
Capturing the old Landlord__c value and creating the amendment record requires two separate flow contexts. The Before Save flow captures the old value (before Salesforce overwrites it). The After Save flow creates the LeaseAmendment__c record. You cannot do both in one flow context.
⚠ New Status picklist on Site__c
The new Status__c picklist (Planned → Fitting Out → Open → Closed) is intended to eventually replace the three existing boolean flags: isActive__c, isComingSoon__c, and Centre_Closed__c. During migration, keep all three booleans active. Only deprecate them once all reports, flows and integrations that reference the booleans have been updated to use Status__c instead.
⚠ Multi-site deals with 3+ sites
Flow 12 fires once per Monthly_Lease__c save. For a 3-site deal with a 37-row import per site, that's 111 flow executions per import run. The governor limit concern is amplified. Test this scenario explicitly before go-live with a real multi-site deal.
⚠ Y1 can have multiple rent schedule rows
The LeaseRentSchedule__c grain is one row per rate period, not one row per year. If a lease has a step-up mid-year (e.g. rent-free for 8 months then full rate), Y1 will have two rows. Reports and Anaplan integrations reading the rent schedule must be built to handle this — don't assume one row per year label.

Layout principles
How fields are organised

Every layout follows the same structural rules. Fields are grouped into collapsible cards. Each card carries a stage badge (Board Approved, Feasibility, etc.) so IM team members know exactly when to fill it in. Field type badges (RAW INPUT, FORMULA, COPIED, SYSTEM, LOCAL TEAM) appear next to every field label so there is never ambiguity about the data source.

Field badge system
What the badges mean
RAW INPUT
IM team types this value directly into the field. It doesn't come from anywhere else and isn't calculated.
FORMULA
Salesforce calculates this automatically. Never manually edited. Examples: Opening Year, Lease Term in Months, Earliest Expiry.
COPIED
Written from the Opportunity by a Flow at a specific stage. Once copied it lives independently — changes on the Opportunity don't auto-sync back.
SYSTEM
Set by a named Flow (Flow 01, Flow 03 etc.) or Salesforce itself. IM team never needs to enter this. Example: Status, Parent External ID, Lease Number.
LOCAL TEAM
Entered directly on the Site__c record by the Local Team after Lease Signed and before the site opens. Never comes from the Opportunity.
ROLL-UP
Aggregated from child records. Example: Total Actual Area (m²) sums Actual_Area_SQM__c across all Active LeaseSite__c records for this site.
